模块 1：Elevator Pitch

针对问题："Tell me about yourself"
⚡ 教练备注（朗读前请先看）
  • 目标时长：90–120 秒。整体结构是：身份锚定 → 专长亮点 → 两段经历各一句高光 → 为什么在这里。
  • 不要背课文，要像在跟一个聪明的朋友解释你的工作一样说话。
  • 第一句是钩子，务必自信说出来。

Well, / the short version is — // I'm a statistician and data scientist / who specialises in figuring out why things happen, / not just what happened. //

I've spent the last three years at two of China's biggest tech companies — ByteDance and DiDi — / and my core focus has always been causal inference and experimental design. // You know, in fast-moving consumer tech, / everyone's got dashboards full of numbers. / But the hard question — / the one that actually moves the needle — is: / did our intervention cause this outcome, / or did it just coincide with it? // That's the problem I wake up thinking about. //

At ByteDance, / I was embedded in the Toutiao and short-story content teams. / I designed and ran rigorous A/B testing frameworks — / we're talking full statistical power calculations, non-inferiority evaluations, confidence interval controls — / and delivered results like a 3.9% reduction in peak-hour bandwidth costs / without any statistically significant drop in user engagement. // So the business saved real money, / and we had the numbers to prove it was safe. //

Then at DiDi, / I moved into a more complex space. // I mean, / ride-hailing is a much noisier environment — / multiple products, overlapping experiments, macro fluctuations. / My biggest challenge there was a platform-wide crisis: / a major homepage redesign that was silently cannibalising our core ride-hailing DAU and GMV. // I used Difference-in-Differences modelling to isolate the causal mechanism — / it turned out invasive UI pop-ups were redirecting users and trapping them in the wrong product flow. // On top of that, / I actually built an LLM-powered pipeline to automate the statistical analysis — / SRM checks, variance decomposition, dynamic trend engines — / so we could iterate much faster than a manual workflow would allow. //

What's really interesting about my background, / I think, / is that I sit at this intersection / of rigorous statistical methodology / and pragmatic engineering — / I don't just build models in isolation, / I build systems that actually change business decisions. //

And honestly? // Australia's data science market — / particularly companies like Canva or Atlassian that are genuinely data-driven — / is exactly the environment where I think this skillset has the most impact. / So / I'm really excited to be having this conversation. //

模块 2：Project Deep-Dive A

针对问题："Tell me about a time you solved a complex problem"  /  "Walk me through your most challenging project"
⚡ 教练备注（朗读前请先看）
  • 目标时长：3–4 分钟（这是面试中最核心的 deep-dive，可以给足时间）。
  • 结构严格走 STAR：危机引入 → 为什么难 → 我怎么分层拆解 → 方法论亮点（DID + LLM）→ 结果 + 推动落地。
  • 记住：面试官想听到的不是"我用了 DID"，而是"我在别人都束手无策的时候，用 DID 找到了真正的原因"——要有戏剧张力。
[SITUATION] 危机引入

So, / let me walk you through what I think is probably my most complex project to date. //

This was at DiDi. // We'd just launched a major homepage redesign — / you know, a full architectural overhaul. / The product team had added a new top-navigation framework / to promote DiDi's freight business alongside its core ride-hailing. / Big strategic bet, / a lot of stakeholders invested in it. //

And then, / almost immediately after rollout, / we saw something deeply alarming. // Our platform-wide DAU dropped 0.11% / and GMV dropped 0.24%. // Now, / I know those numbers might sound small — / but at DiDi's scale, / that translates to millions of dollars in lost transaction volume. // And here's the thing: / the business also thought the freight numbers were looking great, / so there was genuine pressure to call this a success. //

The problem was — // nobody could tell if the ride-hailing losses were real, / or if it was just noise. //

[TASK] 为什么这个问题特别难

And this is where it got really tricky. //

In a normal A/B test scenario, / you'd just compare your treatment and control group, / and you're done. // But this wasn't a clean experiment. / We had multiple overlapping interventions happening simultaneously — / different freight promotion strategies running in parallel, / different user segments, / different traffic sources — / all mixed together in the same traffic pool. //

So the question wasn't just "did the new homepage hurt ride-hailing?" / The real question was: / "causally, / what specific mechanism is driving users away from completing a ride?" // And you can't answer that with a simple dashboard. //

[ACTION] 分三层拆解：实验设计 → DID → LLM 自动化

So I designed a three-layer investigative approach. //

Layer one was pure experimental architecture. // I restructured the traffic allocation — / making sure the ride-hailing and freight experiments were properly orthogonal, / using the same hash seed across app and mini-program traffic / so we weren't comparing apples and oranges. // This gave us clean, isolated treatment effects for each sub-experiment. //

Layer two is where I went deeper. // Even with the A/B data, / I still couldn't rule out macro confounders — / you know, seasonal fluctuations, / external market shifts, / things that affect all users regardless of which group they're in. // So I applied a Difference-in-Differences model. //

The idea was simple but powerful: / I used the freight users as a kind of control reference / to benchmark what "normal" ride-hailing behaviour should look like / in the absence of the UI change. // And once I stripped away all those macro effects, / what I found was crystal clear. // The causal mechanism was a specific UI pop-up — / the one that was aggressively redirecting ride-hailing users into the freight flow. / And crucially — / once users landed in the freight section, / they couldn't find their way back to the ride-hailing homepage easily. // That was the churn node. / I quantified the causal loss at every step of the funnel using a sequential substitution algorithm — / essentially a Shapley-value decomposition of the conversion chain. //

Now, / here's the part I'm actually quite proud of — / Layer three. // The depth of causal analysis I just described? / It would normally take weeks. / You've got SRM checks, variance analysis, trend decomposition — / each one is a manual workflow. // So I built an LLM-powered automation pipeline. / I engineered a suite of statistical agents — / programmatic components that could automatically run A/A validation, / check for sample ratio mismatch, / decompose variance, / and generate dynamic trend engines — / all triggered on demand. // It cut our analysis turnaround time dramatically, / which meant the business could iterate on UI fixes in days instead of months. //

[RESULT] 结果与落地

And the outcome was really satisfying. //

We presented the DID findings with what I'd call incontrovertible statistical evidence — / not "we think it's probably this," / but "here is the precise causal pathway, / here is the quantified loss at each funnel node, / and here is the confidence interval." //

The product team moved fast. / They rolled out a series of rapid UI iterations — / business-specific visual differentiation, / adjusting the pop-up trigger timing, / adding a top-navigation back-to-ride cue. // And the losses were halted. //

But more broadly? // I think what made this project meaningful / was that it changed how the team thought about measurement. // Before, / the instinct was to look at aggregate metrics and draw conclusions. // After this, / there was a much deeper appreciation for the question of causality — / why something happened, / not just that it happened. // And that shift in analytical culture / is honestly the outcome I'm most proud of. //

模块 3：Project Deep-Dive B

针对问题："How do you measure impact when A/B testing is tricky?"  /  "Tell me about a time you had to go beyond standard A/B testing"
⚡ 教练备注（朗读前请先看）
  • 目标时长：2.5–3.5 分钟。这个故事的核心卖点和模块 2 不同——模块 2 展示的是"我能找到问题在哪"，模块 3 展示的是"我能精确量化增量价值，然后把它变成商业策略"。
  • 技术亮点有三层：HTE 框架设计 → Uplift 拆解（真实增量 vs 自然转化）→ 对冲策略落地 + 数字结果。
  • 叙事节奏：先抛出"A/B testing 不够用"的难题，再一层层揭示你怎么解决的。
[SITUATION] 背景：A/B 测出正向，但问题没那么简单

So, / this is actually one of my favourite examples to talk about — / because it's a case where the A/B test gave us a positive result, / but if we'd stopped there, / we would've made a completely wrong business decision. //

The context is DiDi's ride-hailing business. // We had a promotional strategy — / I'll call it a "re-engagement intervention" — / designed to bring back lapsed users who'd drifted towards competitors. / The mechanics involved targeted outreach: / push notifications, / SMS, / and a "shake to get a coupon" mechanic / inside the app and across external mini-program ecosystems. //

The A/B test came back green. // Overall transaction volume was up. / The business team was excited. / And then someone asked the question / that completely changed the direction of the analysis: //

"But wait — / of all the users who redeemed that coupon / and placed a ride… / how many of them would have ridden with DiDi anyway?" //

That is the incrementality problem. // And it's one of the hardest things to get right / in consumer tech experimentation. //

[TASK] 为什么这个问题特别难

Here's why standard A/B testing falls short in this scenario. //

When you run a promotion on a population / that already has a high baseline intent to convert, / what you're measuring isn't incremental value — / you're measuring subsidised behaviour. // You're essentially paying for rides / that would have happened anyway. / I mean, / that's not impact — that's just cost. //

And in a competitive ride-hailing market / where you have users simultaneously active across DiDi's App, / WeChat mini-program, / and Alipay mini-program — / the user behaviour is incredibly heterogeneous. // Different user segments respond completely differently to the same intervention. // So we needed a framework / that could separate the true uplift from the noise / at a granular level. //

[ACTION] 三层方法论：正交 RCT → 用户分层 → HTE Uplift 拆解

So I designed what I'd call a multi-layered orthogonal RCT framework. //

First, / I restructured the experimental architecture. // The key challenge was that the App and mini-program ecosystems / were essentially competing for the same users. / So I made sure the experiments across these traffic channels / were orthogonally designed — / using a shared hash-seed allocation / so the same user couldn't simultaneously appear in conflicting treatment arms. // This gave us a clean, uncontaminated counterfactual baseline. //

Second, / I introduced cohort stratification / based on two dimensions: / user LTV — their long-term value to the platform — / and user state — specifically, / whether a user was a wobbler / who genuinely switches between DiDi and competitors depending on incentives, / or a loyal high-frequency user / who rides with DiDi regardless. // This stratification is critical, / because the business value of winning back a wobbler / is fundamentally different / from the cost of subsidising a loyal user. //

Third, / and this is the core — / I applied a Heterogeneous Treatment Effects framework / to measure the incremental uplift for each cohort separately. // The key question I was answering was: / for each user segment, / what is the causal lift in transactions / that is attributable purely to our intervention, / above and beyond what would have happened in the counterfactual world / where we did nothing? //

What the analysis revealed was really stark. // For high-frequency, high-LTV users — / the loyal core — / the conversion rate was high, / but the incremental value was near zero. // They were going to ride anyway. / Spending money on them was pure waste. //

For the wobbler cohort — / users in that middle zone / who genuinely oscillate between platforms based on price sensitivity — / the incremental lift was significant and real. // The intervention was actually changing their behaviour. //

[RESULT] 对冲策略落地 + 量化结果

So based on those uplift findings, / I formulated what I called a statistical hedging strategy. //

Instead of running the promotion broadly, / we concentrated the intervention budget on the high-incrementality wobbler segment. / We basically stopped subsidising the loyal users / and redirected that spend / towards the users where every dollar of promotion was actually doing something. //

The quantified outcomes were significant. // We successfully retained 1.27% of high-frequency active users / from competitive siphoning — / users who, without the intervention, / we had strong statistical reason to believe / would have shifted their ride volume to a competitor. //

And over a two-month observation window, / the full strategy rollout achieved a +6.03% statistically significant causal lift in core App transaction volume, / and a +6.6% increase in App transacting users. // These aren't correlation numbers — / they're causal estimates / derived from the counterfactual baseline / I'd built into the experimental design from day one. //

What I'm proud of here / isn't just the number — / it's the reasoning chain that got us there. // We went from / "the A/B test is green, let's scale it" / to / "actually, let's ask who is responding, / and why, / and whether that response is genuinely incremental." // That shift in analytical rigour / is what turned a good-looking experiment / into a genuinely high-ROI business decision. //

模块 4：Behavioral & Culture Fit

两个最高频问题的逐字稿
⚡ 全局教练备注
  • 模块 4 和前三个模块的底层逻辑完全不同。前三个是"展示能力"，这个是"展示人格"。
  • 澳洲外企文化极度重视：主动性、透明度、心理安全感、以及不 PUA 的成熟职场关系。
  • 你的回答要让面试官感受到：你是一个有自我认知、有方向感、且离职原因正向的人——而不是一个在逃跑的人。
问题 A："Why do you want to leave your current role?"
⚡ 问题 A 教练备注
  • 绝对禁区：不提移民、不提"苟着"、不抱怨管理层。
  • 核心叙事策略：主动向前奔跑，而不是被动逃离。
  • 把离职理由包装成三个维度：① 技术深度的天花板 ② 对国际化协作环境的渴望 ③ 在澳洲市场找到更好的专业匹配度。

That's a fair question, / and I want to give you an honest answer. //

I've genuinely learned a lot at DiDi. // The scale is enormous, / the problems are complex, / and I think some of the work I've done there — / the causal inference frameworks, / the LLM-powered pipeline — / I'm really proud of. //

But to be honest with you? // I've reached a point where I feel like I've extracted most of the learning I can from this particular environment. // You know, / in Chinese big tech, / the culture is intensely execution-focused — / it's very fast, / very competitive, / and the emphasis is on shipping and iteration. // And I've thrived in that. / But what it sometimes lacks / is the space for deeper methodological dialogue — / the kind of conversations where you're genuinely debating / whether DID is the right causal model here, / or whether we need an IV approach instead. // I want to be in a room / where that rigour is the default, / not the exception. //

The second thing — // and this is probably the bigger pull factor — / is that I'm genuinely excited about working in an international environment. // The problems that companies like yours are working on — / data-driven product decisions at global scale, / with diverse, cross-functional teams — / that's exactly the kind of work / where I believe my particular combination of skills / has the most impact. //

So it's less about leaving something behind, / and more about deliberately moving toward something / that I think is the right next chapter for where I want to take my career. //

问题 B："Tell me about a time you faced pushback on your analysis. How did you handle it?"
⚡ 问题 B 教练备注
  • 这道题考察的不是你有没有遇到阻力，而是：你如何在坚持数据立场的同时，维护职场关系。
  • 最佳答案结构：描述冲突 → 我用数据而非情绪回应 → 最终赢得信任 → 反思。
  • 用 DiDi 首页重构那个故事——因为你当时的分析结论（新首页有负向损失）是反商业直觉的，业务侧肯定有压力。

Yeah, / this actually comes up more than you'd think / in a fast-moving environment like DiDi. // Let me give you a specific example. //

This was during the homepage redesign project I mentioned. // When I first brought the data showing that the new homepage was causally hurting ride-hailing metrics, / the reaction from the product team wasn't exactly warm. // I mean — / they'd invested months into this redesign, / there was real excitement about the freight business growth, / and here I am showing up with a DID model saying / "actually, / I think we have a problem." //

The pushback I got was essentially: / "Your model might be wrong. / The aggregate numbers still look okay. / Maybe you're overcounting the effect." //

And, / you know, / my instinct in that moment / was to defend my methodology. // But I've learned — / and this took me a while — / that the most effective response to pushback on analysis / isn't to double down on the method. // It's to invite the sceptic into the reasoning process. //

So instead of saying "my DID model is correct," / I said: / "Let's stress-test it together. / Tell me what assumptions you think might be wrong, / and we'll run the sensitivity checks right now." //

We spent time going through the parallel trends assumption, / checking whether the freight user cohort was genuinely a valid control reference, / running robustness checks with different model specifications. // And what happened was — / by the end of that session, / the product manager wasn't just persuaded by the conclusion, / they were persuaded by the process. // They'd seen exactly where the numbers came from. //

The output was that the team moved quickly on the UI iterations I recommended — / not because I'd won an argument, / but because we'd arrived at the conclusion together. //

What I took away from that is something I now try to apply consistently: // statistical rigour and collaborative humility aren't in tension — / they actually reinforce each other. / If your analysis is sound, / it can withstand scrutiny. / And inviting that scrutiny / is the fastest way to build trust. //

Module 1 · ByteDance 追问

Q11 – Q13
Q11
🟡 P1
"For the live-streaming bitrate reduction — how did you calculate the required sample size and statistical power before running the experiment?"

Good question — sample size planning was actually a non-trivial step here, because the primary metric we were protecting was user active days (lifetime), which has a relatively low variance compared to engagement metrics like click-through rate. //

So the approach I took was: // first, I pulled the historical distribution of the metric — active days per user over a 30-day window — to get the baseline mean and standard deviation. // Then I used a standard two-sample t-test power formula: / given a target minimum detectable effect (MDE) — in this case, we set a non-inferiority margin of roughly 0.1% on active days — / with 80% statistical power and a significance level of 0.05, / I calculated the required per-arm sample size. //

The tricky part was that we were running the experiment on a subset of low-value users — / not the full user base — / so I had to re-estimate the variance within that specific cohort separately, / because low-engagement users tend to have much higher variance in activity patterns. //

Once I had the per-arm n, / I cross-checked against the actual daily traffic volume in that segment / to determine how many days the experiment needed to run / to reach the required exposure. //

In practice, / we ran it for 14 days / with a holdout buffer of a few extra days for metric stabilisation before readout. //

"You mentioned using freight users as a control group in your DID model. How did you validate the parallel trends assumption? What did you do if it didn't hold?"

Great question — / the parallel trends assumption is the central credibility requirement of any DID design, / and I took it seriously. //

The validation approach was: / I looked at the pre-treatment trend of the ride-hailing metric — / specifically daily active users and transaction volume — / for both the affected group (ride-hailing users exposed to the new UI) / and the control reference group (freight users) — / going back 8 weeks before the homepage redesign launched. //

Now — / you asked what I'd do if it hadn't held. // Honestly, / if the pre-trends had diverged, / I would have abandoned DID for that specification / and pivoted to one of two alternatives: / either a synthetic control approach — / constructing a weighted combination of multiple control groups to better match the pre-trend — / or a regression discontinuity design if the treatment timing was sharp enough. // I actually kept those options in reserve, / because with a redesign launch date as a clean cutoff, / RD was a viable fallback. //

Q2
🔴 P0
"When you say you used Shapley values for funnel decomposition — how exactly did you define the 'players' in that Shapley game? Was it funnel steps, or user segments, or something else?"

Good catch — / I should be precise here. //

The "players" in the Shapley decomposition / were funnel conversion steps, / not user segments. // Specifically, / I defined the funnel as a sequential chain of conversion events: / homepage entry → top-nav click → freight page landing → ride-hailing return attempt → ride-booking completion. //

The question I was answering with Shapley was: / of the total observed drop in ride-booking completions, / what proportion of that loss is attributable to each step in the chain? //

So the "contribution" of each player — each funnel step — / was measured as its marginal contribution to the total conversion loss / averaged across all permutations of which steps you include. // This gave me a principled way to say: / "the largest single contributor to the conversion loss is step 3 — users landing on the freight page and failing to navigate back" — / rather than just eyeballing drop-off rates. //

In practice, / I implemented this as a sequential substitution algorithm — / which is computationally equivalent to Shapley for ordered funnels — / rather than the full combinatorial Shapley calculation, / which would've been expensive on our data volume. // The ordering assumption in a funnel is justified, / so this approximation was valid. //

🔴 P0

The solution was at the allocation layer. // I used a shared hash-seed across both the ride-hailing and freight experiment arms — / meaning a given user's experiment assignment was deterministic based on their user ID and the same seed. // This ensured that the same user couldn't simultaneously be in the treatment arm for ride-hailing / and the treatment arm for freight — / they'd be consistently assigned to one side. //

🔴 P0
"You mentioned SRM checks in your LLM pipeline. What's your typical threshold for flagging an SRM, and what do you do when you find one?"

So SRM — Sample Ratio Mismatch — / is essentially a sanity check that verifies / the actual traffic split between treatment and control / matches the intended split from the randomisation layer. // If you intend 50/50 but observe 52/48, / something is wrong upstream. //

My threshold for flagging: / I use a chi-squared test on the observed vs. expected counts — / and flag any result with p < 0.01 as a hard SRM. // I deliberately use a stricter threshold than the standard 0.05 / because SRM at scale can be subtle — / even a 1% deviation in sample ratio / can significantly bias treatment effect estimates. //

When I find one, / the experiment is paused immediately — / no results are read out until the root cause is identified. // The most common causes I've seen: / logging loss on one variant, / client-side filtering that applies asymmetrically, / or a triggering condition that was accidentally correlated with the hash-based assignment. //

In the LLM pipeline I built, / the SRM check runs automatically at experiment launch — / it fires after 24 hours of data collection / and sends an alert if the ratio is off. // The model generates a structured diagnostic report flagging which traffic source the mismatch is concentrated in, / which usually points directly to the root cause. //

"Can you explain the key assumptions underlying Difference-in-Differences, and describe a situation where those assumptions were threatened?"

The DID estimator rests on four key assumptions: //

One — Parallel trends: / in the absence of treatment, / the outcome of the treatment group would have evolved / in the same way as the control group. // This is the central and most critical assumption, / and it's fundamentally untestable in the post-period — / you can only provide pre-period evidence in its support. //

Two — No anticipation: / subjects in the treatment group do not change their behaviour before the treatment is implemented, / in anticipation of it. //

Three — SUTVA — Stable Unit Treatment Value Assumption: / the treatment of one unit does not affect the outcome of another unit. // In other words, / no spillovers or network effects between groups. //

Four — Common support: / the treatment and control groups are comparable in their observable characteristics. //

Now — / the assumption that was most threatened in my DiDi DID analysis was SUTVA. // The concern was spillover: / if a ride-hailing user who was redirected to the freight page influenced another user's behaviour — / say, through a social referral mechanic — / the control group could be contaminated. //

I addressed this in two ways. // First, / I analysed the geographic clustering of treatment and control users / to check whether treated users were disproportionately clustered in the same social networks or geographic zones as control users. // The clustering was not statistically anomalous — / the randomisation appeared to distribute users across network clusters evenly. //

Second, / I ran a sensitivity analysis that excluded users with any detected social referral link to a treated user. // The DID estimate was robust to that exclusion — / the coefficient changed by less than 5%, / well within the confidence interval. //

That robustness check is what gave me the confidence to defend the SUTVA assumption. //

Third — methodological choice. // When the standard A/B test framework isn't appropriate — / because of contamination, / or confounders, / or the lack of a clean randomisation — / I choose and implement the right method for the problem: / DID, instrumental variables, synthetic control, non-inferiority design. // A Data Analyst doesn't typically have the training to make those methodological choices. //

To put it plainly: / the Data Analyst title in my HR records reflects my organisational classification in a Chinese tech company, / where the title is used broadly. // The substance of the work — / statistical modelling, causal inference, experimental design, uncertainty quantification — / is what a statistician does. //
